学用Egret制作2048的过程

  1. 在Excel中用VBA实现，实际上没有实际做展示，就是处理16个数字的逻辑（主要是移动和合并）。

    • 步骤是先只处理向上移动，然后分析完上下移动的区别之后，增加向下移动。
    • 然后分析水平移动和垂直移动的区别，处理水平移动。
    • 在处理向上时想通逻辑后比较好实现，后来为了避免有四段代码，采用通过修改参数的方式让同一段代码执行四向移动反而是遇到很多困难，但是这一步是有收益的，因为后来这段代码有很多问题并且被改了几次。

  2. 用Egret实现最基础的版本（绘制版），学习Egret的函数、方法，以及TS的语法消耗了比较多的时间。

    • 绘制版主要分两个部分 4x4数组（逻辑层） 和 根据数组每个位置的数字绘制自己（展现层）。
    • 逻辑层完全是从VBA翻译过来的。
    • 展现层是采用的graphic的方式实现的，结果发现特别的卡，引入cacheAsBitmap之后发现格子绘制不全，未找到解决办法。
    • 在处理触摸移动时，是在抬起手指时做的判断。

  3. 实现基础版本之后主要是处理流畅度，第一部分包括：

    • 在手指抬起时再判断移动方向是体验差的最主要原因，修改成在手指移动过程中就判断距离之后，流畅度有了第一次提升，也是最大幅度的提升。
    • 不停的重新graphic导致手机非常热，修改成为图片，流畅度和发热都有好转。
    • 将图片打包为一张整图，这里并没有明显感知到优化。

  4. 试图增加动画：

    • 增加渐入渐出，这个长期没有成功，因为很脑残的是1.加载图素，2.将资源的透明度降低到0，3.将资源透明度渐增到1，其实应该将1、2步骤对调。
    • 增加移动动画，长期卡在这里，遇到的问题很多，一直没有进展。
    • 去掉底板，由于desktop不仅是所有元素的容器，同时还是触摸监控器；去掉deskop的background之后，导致了只有在触摸在格子上时才能被监控的bug。

  5. 尝试显示总分，遇到的问题有：

    • 设置cookie或者是调用egret自己的函数时，都需要将数字转化为文本储存。同样的，读取到的信息也全部是文本。
    • 未初始化的数字，在转化为String时会报错，这个报错查了非常久，因为报错的位置不是在自己写的代码中，而是在库里面，当时以为是库有问题。
    • 第一版实现的总分是屏幕上的所有数字和，但是原版实际上是每次合并时，就把合并后的数字增加到总分中，游戏感觉完全不一样。

  6. 尝试：

    • 拆分Hello2048（触摸和主逻辑）、Grid（展现层）、Data（逻辑层）三个部分，这个最开始只有Data独立出来了但是里面几乎没有逻辑。
    • 尝试将UI绘制独立出来，一直没有成功，因为在绘制UI过程中会使用到太多Hello2048中的变量和数据。
    • 尝试将游戏结束时的 alert 修改为 GUI.alert 最终没有成功。

  7. 移动动画中遇到的问题和经验：

    • 循环变量不要随便复用，两个for循环都使用的 i，j 作为变量，有坑，没必要在申请变量时偷懒。
    • for 循环和 if 嵌套太多导致代码不可读，第一个bug就是本应该在外层的 for 循环被写到了前一个 for 循环里面了。
    • 拦挡时需要注意，遇到的问题是在回收被合并的单元格时，当时的处理逻辑是：
      • 逻辑层先将这个单元格设置为未被使用和被删除
      • 『单元格回收』是在『单元格移动』之后，否则这个单元格就没有动画直接消失了。
      • 未被使用的单元格是没有位置信息的，处理移动时会报错。
      • 于是在『单元格移动』之前，增加了判断，未使用的单元格不进入『单元格移动』逻辑。
      • 第2条和第4条联合导致了单元格无法被回收：因为被合并的单元格先被标志为被删除 -> 无法进入『单元格移动』逻辑 -> 无法进入『单元格回收』逻辑
    • 解决了前两个问题之后，遇到了时序的问题：
      • 仍然是因为需要『在动画结束之后再回收单元格』。
      • 问题是逻辑层在动画结束前就已经在寻找新单元格了，此时找到的单元格尚未被回收，于是报错。
      • 还有个问题假如用户在动画结束之前就再次输入，可能会导致逻辑层和表现层不一致，这个最终是限制300ms（动画播放时间）内只能输入一次。
      • 最初输入拦挡的启动是在触摸输入开始时，而开启计时器确是在输入成功后（手指移动超过6px），导致了假如只单击就会游戏卡死，又调整了一版。

  8. 之后是增加了UA判断：

    • 背景是游戏内嵌到了觅题中，而Android是App来控制游戏是否结束（为了分享），而iOS没有这个设计，之前一直是做两个版本。
    • 后来通过UA来判断用户的设备信息，一个版本两端都可以使用。
    • 目前找到的判断UA的函数有一堆报错，不过将就可以使用。

  9. 过程中的一些其他问题：

    • Egret自带服务器命令，但是当时自己不知道build之后服务器会自动更新，也不知道 ctrl+c 可以关闭服务器，后来从官方QQ群中学到。
    • 发布：
      • 发布到SAE，有教程基本没有费事儿。
      • 游戏内嵌到觅题之后，访问量暴高，于是首先上传到了CMS，由于CMS只能传静态文件，也在此时得知index.php其实是没有用上的（至少我的游戏是）。
      • 后来学会了上传到Github，并且通过 Markdown 做了一个统一的入口页。
    • 数据统计：
      • 在嘿嘿的帮助下接入了百度统计和谷歌分析，后来就只用百度统计了，这个接入需要修改 index.html 。
      • 尝试了一下接入Dataeye，发现会比较简单一些（不用每次都在 index.html 中添加统计代码），但是没有使用（库有点大）。
      • 后来发现Egret发布游戏的时候，主要修改的是game-min.js，没有修改 index.html ，于是百度统计也变得简单了。

      • 感受了一下，容器不会用：某些容器没有percentWidth等属性，某些容器不能add带percentWidth的属性作为Child或者Element。
